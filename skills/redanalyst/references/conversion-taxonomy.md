# Таксономия конверсий — не путать классы

Самая частая ошибка новичка — смешать клиентские (JS) цели и серверные (офлайн) конверсии. У них
разная механика, разные ограничения, разные способы отладки. Держи их раздельно.

## 1. Клиентские цели / воронка (reachGoal, JS) — real-time

- Срабатывают в браузере, **пока пользователь идёт по шагам**. `ym(id,'reachGoal','order_click')`.
- **Нет «окна отправки»**, не зависят от бэка/крона. Видны в отчёте почти сразу.
- Типы целей (Management API `type`):
  - `url` — посещение страницы (URL совпал с условием). Пример: `/thank-you`.
  - `action` — JS-событие (`reachGoal` с идентификатором). Пример: `order_click`, `form_submit`.
  - `number` — число просмотров страниц за визит.
  - `step` — **составная цель = воронка** (упорядоченные шаги; смотришь, где отваливаются).
  - `ecommerce` — покупка через dataLayer (см. `ga4-gtm.md` для структуры товаров).
- **Micro vs macro** — ставь ОБЕ:
  - *micro* (промежуточные намерения): просмотр карточки, добавил в корзину, начал оформление, клик
    по телефону. Дают раннюю оптимизацию и объём для статзначимости.
  - *macro* (целевое действие): отправил заявку, оплатил. Их мало, но это деньги.
- **«Цель создана» ≠ «цель срабатывает».** После создания action-цели проверь реальное срабатывание
  (сделай событие на сайте / посмотри отчёт по цели за сегодня). Частая причина «не срабатывает»:
  неверный тип vs реальный триггер, несовпадение имени JS-события, счётчик не на той странице.

## 2. Офлайн-конверсии (серверные) — для «оплата известна только на сервере/после»

Когда факт продажи возникает НЕ в браузере: оплата подтверждается эквайером/в 1С/в CRM, звонком,
в оффлайн-точке. Тогда конверсию **досылают на сервер Метрики**.

- Загрузка: `POST /management/v1/counter/{id}/offline_conversions/upload` (Management API), CSV.
- **Требуют идентификатор** (≥1, иначе строка не привяжется → LINKAGE_FAILURE):
  `ClientId` (самый точный) | `UserId` | `Yclid` | `PurchaseId`. Без идентификатора привязать продажу
  к визиту **невозможно** — это не «плохо привяжется», а «никак».
- **Окно = 21 день.** Событие старше 21 дня к визиту не приклеится. Сначала спроси у API реальный
  порог: `GET .../offline_conversions/visit_join_threshold` — окно открывается постепенно после
  включения опции. Не эмить строки старше порога.
- `Target` в CSV — это id **action-цели** (JS-события), заведённой заранее под офлайн (напр.
  `order_confirmed`). `DateTime` — unix-время, обязано быть в прошлом. `Price`/`Currency` опциональны.
  Файл **UTF-8**.
- **Revenue = фактически оплаченная сумма** (с учётом промокода/частичной оплаты), а НЕ намерение и не
  сумма корзины. Возврат/частичная оплата → не считать полной (см. `offline-matching-recipe.md`).
- Счётчик должен иметь включённую «Загрузку данных» (Настройка счётчика) — ручной prerequisite.

## Как выбрать, что ставить

1. Есть ли шаг в браузере, который и есть целевое действие (заявка/оплата картой на сайте)? → JS-цель.
2. Оплата подтверждается вне браузера (1С/эквайер/CRM/звонок)? → офлайн-конверсия + захват ClientID
   при создании заказа (см. `clientid-capture.md`).
3. Всегда: micro-цели воронки (JS) + macro (JS или офлайн, смотря где истина об оплате).

> Урок кейса: сайт показывал «0 офлайн-конверсий», а продажи были — просто оплата жила в 1С, а не
> в браузере, и без ClientID её нельзя было приклеить к визиту. Класс проблемы — «настроено, но не
> течёт»; ловится только сверкой с источником истины (см. CORE PRINCIPLE в SKILL.md).
