# Кейс-эталон — e-commerce на Bitrix/1С с эквайрингом (на нём родился скилл)

Обобщённый реальный кейс: интернет-магазин на Bitrix, эквайринг банка через 1С (оплата картой
подтверждается на стороне банка/1С, не в браузере). Показателен тем, что каждая грабля здесь —
типовой класс, а не разовая случайность. Это эталон для аудита.

## С чего началось

Вопрос звучал бытово: **«а были ли вчера продажи и видит ли их Метрика?»** Метрика показывала
**0 офлайн-конверсий**. Наивный вывод — «продаж нет / аналитика сломана». Правильный ход — не поверить
прокси, а пойти к источнику истины.

## Что нашли (расследование сверху вниз)

Два **независимых** блокера + третий позже:

1. **Сайт вообще не забирал факт оплаты.** Эквайринг был через банк/1С, а код подтверждения оплаты
   написан под **другого платёжного провайдера** (частая история после смены эквайера). Статус
   `SUCCESS` не выставлялся **никогда** → в выгрузку офлайн-конверсий попадали нули. Классика:
   «настроено под другой платёжный провайдер».
2. **ClientID терялся в whitelist контроллера заказа.** Даже когда оплата была, привязать её к визиту
   было нечем — ClientID отбрасывался промежуточным слоем и не доезжал до записи заказа.
3. **Крон-хост не был в планировщике.** Скрипт выгрузки работал руками, но по расписанию не
   запускался — крон-задача жила (точнее, не жила) в панели хостинга, а не в `crontab -l`.

**«0 SUCCESS ≠ 0 продаж».** Продажи БЫЛИ — подтвердили **live-запросом к 1С `purchase_history`**:
реальные оплаты картой. Сайт просто про них не знал.

## Что сделали

- Ушли от ненадёжного «поллинга эквайера» к чтению **1С `purchase_history`** (доступно из крона по
  персистентному usertoken).
- Матчинг по **времени + сумме + `payments`**; завели **отдельное поле** под YM-статус — не трогая
  платёжный статус заказа (он гейтил формирование ссылки на оплату, менять его было нельзя).
- Досылали офлайн-конверсии в Метрику.

## Дисциплина (как доводили)

- **2× plan-panel** до кода.
- **finalize** с харденингом: окно матча **±3с**, **comma-decimal** нормализация сумм,
  **идемпотентность** по id покупки, **dry-run**, **kill-switch**, **маскировка токена**.
- Раскатка **dry-run → canary → live-verify**.

## Результат

Сквозная фиксация продаж заработала (конверсии с revenue в отчёте). Оптимизировано под нагрузку 1С:
опрашивали **только заказы с ClientID** → **−95% запросов**.

## Два урока, вшитые в скилл

- **Урок №1 (ядро аудита):** самый частый и незаметный класс — **«пайплайн настроен, но НЕ течёт»**.
  Аудит обязан его ловить, а ловится он только сверкой с источником истины, не проверкой галочек.
  Отсюда CORE PRINCIPLE «настроено ≠ работает».
- **Урок №2 (не тратить время):** **backfill истории невозможен без ClientID.** Окно 21 день + у
  старых заказов ClientID нет = 0 конверсий, сколько ни грузи. Не обещать заказчику «поднимем всю
  историю продаж в сквозную» — это физически нельзя. Захват ClientID начинает работать только вперёд.
