# Сквозная аналитика = 3 ноги. Нет ноги → нет ROI

Сквозная (end-to-end) аналитика отвечает на «сколько заработали с рубля рекламы по каждой кампании».
Для этого нужно свести три независимых потока данных на общий ключ. Убери любую ногу — ROI
не посчитается. Большинство спотыкается на третьей.

```
        (а) РАСХОД            (б) ВИЗИТЫ           (в) ВЫРУЧКА
     сколько потратили     кто пришёл и откуда   кто реально заплатил
     ────────────────      ─────────────────     ──────────────────
     Директ↔счётчик        yclid / UTM           purchase-цель +
     Costs import API      автопометка           офлайн по ClientID
            \                    |                      /
             \___________________|_____________________/
                         ключ: визит (ClientID / yclid)
                                  ↓
                    ROI / ROMI / CPA по кампании
```

## (а) Расход на рекламу

- **Яндекс.Директ**: указать № счётчика Метрики в настройках кампании. Дальше расход подтягивается
  автоматически; отчёт Метрики «Источники, расходы и ROI» считает ROI сам. Через Reporting API расход
  тянется параметром `direct_client_logins`.
- **Не-Яндекс источники** (VK, Telegram Ads, таргет): нет автосвязки → **Costs import API**
  («Расходы») — грузишь расход по UTM, чтобы ROI был единым по всем каналам.

## (б) Разметка визитов

- **yclid** — автопометка Директа. Включена всегда, отключить со стороны Директа нельзя. Это fallback,
  который склеивает клик по объявлению с визитом Метрики, даже если UTM потерялись.
- **UTM** — для всех прочих источников. Единая UTM-стратегия (source/medium/campaign/content/term),
  консистентная между рекламными кабинетами и разметкой ссылок. Кривые/несогласованные UTM = каналы
  не сходятся, ROI размазывается по `(not set)`.

## (в) Выручка / конверсии — где все спотыкаются

- Нужна **цель purchase** (JS ecommerce ИЛИ офлайн-конверсия) + **привязка оплаты к визиту**.
- Если оплата на сайте (браузер знает факт) → ecommerce-событие, real-time.
- Если оплата вне браузера (1С/эквайер/CRM/звонок) → **офлайн-конверсия по ClientID**. И вот тут
  цепочка рвётся чаще всего:
  - ClientID не захвачен при создании заказа → нечем привязать (см. `clientid-capture.md`).
  - Захвачен, но потерян в промежуточном слое (whitelist контроллера) → в 1С/CRM его нет.
  - Захвачен и доехал, но событие старше 21 дня → окно закрыто.
  - Загружено, но сайт вообще не знал факт оплаты (эталонный кейс: код подтверждения под другой эквайер) →
    в источник выгрузки попадали нули.

## Precondition-guard (решение panel #7)

Не считать ROI/ROMI на **неполных** данных — это хуже, чем не считать вовсе (ложная уверенность):
- нет расхода по каналу → пометить канал «расход неизвестен», не показывать ROI как посчитанный;
- цель purchase не верифицирована (не срабатывает) → выручка недостоверна → блок ROI с явным warning;
- офлайн-конверсии в статусе не-`processed` → выручка частична → показать «данные догружаются, ~2ч».

Табличный вывод crosschannel: колонки `кампания | расход | визиты | конверсии | выручка | CPA | ROMI`
+ явные состояния ячеек (`—` неизвестно / `⏳` догружается / число). Никогда не подменять неизвестное нулём.
