# Матчинг офлайн-продажи к визиту/заказу — рецепт

Ситуация: оплата подтверждается на стороне CRM / эквайера / 1С, и её надо приклеить к веб-визиту и
досдать в Метрику как офлайн-конверсию. Ниже — как делать это надёжно, со всеми граблями, что реально
стреляют на проде.

## Ключ матча

`ClientID заказа` (первичный) **+** `время` **+** `сумма` (**+** стабильный `purchase_id` для
идемпотентности). ClientID отвечает «чей визит», время+сумма подтверждают «та самая оплата»,
purchase_id — «не задвоить».

## Что обязательно обработать

- **Неоднозначность.** Несколько оплат в окне от одного ClientID → НЕ угадывать, какая «та». Матчить
  по стабильному purchase_id/номеру заказа, а не по «ближайшей по времени».
- **Возврат / частичная оплата.** Сумма платежей ≠ total заказа → не считать полной. Revenue =
  фактически прошедшие платежи (сумма `payments`), с учётом возврата. Возврат после конверсии —
  отдельный вопрос (Метрика не умеет «отменить» офлайн-конверсию тривиально; фиксируй политику).
- **Clock skew.** Время оплаты в 1С/эквайере и время визита не совпадут до секунды. Матчить по **окну
  ±N сек** (в кейсе-эталоне ±3с), не по точному равенству.
- **Локаль-формат сумм.** `1 234,56` (запятая-разделитель, пробел-тысячи) → **нормализовать в float**
  ПЕРЕД сравнением/отправкой. Классический баг: `float("1 234,56")` падает или даёт мусор.
- **Идемпотентность — обязательна.** Не задваивать конверсию/revenue при ретрае. Дедуп на уровне
  HTTP-вызова через `lib/ledger.py` (`purchase_id UNIQUE`): запись в ledger ДО upload; ретрай видит
  запись и не шлёт повторно. Двойная загрузка = двойная выручка в отчёте = враньё в ROI.

## Раскатка (как в кейсе)

`dry-run` (собрать CSV, показать, НЕ слать) → `canary` (1–2 строки, дождаться
`GET uploading/{id}=processed` + факт в отчёте) → `live` (полный объём) → `read-back` (сверить число
`processed` с числом отправленного по ledger). **kill-switch** — одна переменная гасит отправку.

## Нагрузка на CRM/1С (важно на проде)

Опрашивать источник истины ТОЛЬКО по заказам, **способным дать конверсию** — у которых есть ClientID
и они в пределах окна. В кейсе-эталоне фильтр «только заказы с ClientID» срезал **~95% запросов** к 1С.
Не тащить всю историю каждый прогон — только дельту с прошлого успешного `batch_id` из ledger.

## Где брать факт оплаты (источник истины)

- **1С**: если есть персистентный usertoken клиента — читать `purchase_history` из крона (в кейсе-эталоне
  так и сделали, уйдя от ненадёжного «поллинга эквайера»).
- **CRM**: API сделок со статусом «оплачено».
- **Эквайер**: реестр платежей (менее удобно — часто нет ClientID/номера заказа на их стороне).

Определи это на аудите (`source_of_truth` в `state.json`) и НЕ доверяй промежуточным прокси
(«статус заказа на сайте» может врать — см. кейс-эталон).

## Матч-алгоритм (псевдокод)

```
для каждого paid_order из источника истины (дельта, только с ClientID, в окне):
    if ledger.has(purchase_id): continue            # идемпотентность
    revenue = normalize_sum(sum(order.payments))    # локаль → float, учесть возвраты
    row = {Target: GOAL_ID, ClientId, DateTime: order.paid_at_unix, Price: revenue, Currency: 'RUB'}
    if DateTime older than visit_join_threshold: skip + log 'outside-window'
    buffer.append(row); ledger.reserve(purchase_id)  # запись ДО upload
upload(buffer) → batch_id → ledger.mark(batch_id, uploading_id)
poll uploading/{id} until processed → ledger.mark(status)
```
