# 152-ФЗ при передаче ClientID + сумм Яндексу

Передача `ClientID` + суммы покупки третьему лицу (Яндексу, при загрузке офлайн-конверсий) — это **акт
обработки персональных данных**. Нужен правовой базис/согласие ДО первой загрузки. Это **комплаенс-
гейт, а не технический** — Метрика не вернёт ошибку за его отсутствие, поэтому легко пропустить.

## Что требуется

- **Раскрытие в политике/оферте.** Факт передачи ClientID+сумм в Яндекс.Метрику должен быть описан в
  политике конфиденциальности / пользовательском соглашении сайта. Не «где-то про cookies вообще», а
  конкретно про передачу данных о покупке в аналитику.
- **Согласие ссылается на реальный документ.** Галка согласия при оформлении заказа должна вести на
  документ, где это **действительно описано**. Классический разрыв: галка есть, документ есть, но в
  документе про передачу в Метрику ни слова → согласие юридически пустое.
- **Проверка frontend ↔ контент.** На аудите: (1) есть ли галка/согласие на форме заказа; (2) на какой
  документ она ссылается; (3) есть ли В ЭТОМ документе пункт про передачу данных покупки в аналитику.
  Все три должны сойтись.

## Тайминг согласия vs getClientID (решение panel #8)

- Согласие должно быть получено **до** передачи данных Яндексу (до загрузки офлайн-конверсии), а в
  идеале — до захвата/использования ClientID для этой цели.
- Практически: getClientID технически срабатывает при загрузке страницы (это работа счётчика, который
  сам по себе — отдельный вопрос cookie-согласия), но **связывание ClientID с суммой покупки и
  отправка в Метрику** — вот тот шаг, который гейтится согласием на обработку ПДн о покупке.

## Переиспользовать consent-kit из redloft

Не изобретать: в redloft есть под-модуль **consent-kit** (152-ФЗ система согласий для сайта,
Bitrix+Next.js). Для redanalyst он покрывает и этот кейс — при setup/ecom, где включается передача
данных в Метрику, направлять на `/redloft-consent` для развёртывания/проверки согласий. redanalyst
лишь **фиксирует требование** и проверяет наличие, а разворачивает согласия consent-kit.

## Вывод для скилла

- В `/redanalyst-audit`: пункт «152-ФЗ согласие на передачу ClientID+сумм» в gap-report (severity:
  зависит — при уже идущей передаче без согласия это **critical** правовой риск).
- В `/redanalyst-ecom` / `/redanalyst-setup`: **hard-gate** — не включать боевую загрузку офлайн-
  конверсий с суммами, пока согласие не подтверждено (или явно не принято решение заказчиком под его
  ответственность, зафиксированное письменно).
