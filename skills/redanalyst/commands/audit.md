# /redanalyst-audit <домен | counter_id> — что подключено и РАБОТАЕТ

Вес: lightweight (read-only). Токен нужен только `metrika:read` (или вообще инструкции человеку, если
доступа к счётчику нет). Всегда стартовая точка — даже если просят «настрой».

## Принцип
Аудит идёт **от данных вниз**, не от галочек. Каждый пункт подтверждается ФАКТОМ (см. CORE PRINCIPLE
и `_shared.md §4`). «0 в отчёте» → иди к источнику истины, не делай вывод «продаж нет».

## Шаги
1. `PD=$(bash lib/persist.sh <slug>)`; создать/прочитать `state.json`.
2. Определить counter_id (из аргумента, из HTML сайта — тег `ym(`, или спросить). Записать в state.
3. **Инвентаризация (read API `metrika:read` или ручной осмотр):**
   - Счётчик стоит и шлёт хиты? (визиты за сегодня в Reporting API / тег на всех ключевых страницах).
   - Цели: список `GET goals`. Для каждой — **срабатывает ли** (`ym:s:goal<id>reaches` за 7–30д). Цель
     без срабатываний = красный флаг (создана ≠ работает).
   - Ecommerce/выручка: есть ли ecommerce-данные / purchase-цель.
   - Директ связан? Виден ли **расход** в «Источники, расходы и ROI» (не «№ вписан»). См.
     `references/direct-linkage.md`.
   - Офлайн-конверсии: включена «Загрузка данных»? Идут ли `uploadings`? Статусы? Диагностический
     построчный отчёт (LINKAGE_FAILURE-причины). См. `references/metrika-api.md`.
   - UTM-разметка: консистентна? Много `(not set)`?
   - 152-ФЗ: есть ли согласие на передачу ClientID+сумм, ведёт ли на реальный документ
     (`references/privacy-152fz.md`).
   - **Источник истины по оплатам**: где реально фиксируется оплата (1С/CRM/эквайер)? Это ключ к
     «настроено vs течёт». Записать `source_of_truth` в state.
   - Крон (если есть офлайн-выгрузка): подтверждён в планировщике хоста? (`references/ops-gotchas.md`).
4. **Сверка «настроено vs течёт»**: если отчёт показывает мало/ноль конверсий — сверить с источником
   истины (сколько реально продаж за период). Расхождение = класс «настроено, но не течёт» → это
   главная находка.
5. **gap-report** (`gap-list.md` + `state.gap_list`): каждый пробел с `severity` (critical/warning/
   info) и коротким «почему важно» + «как чинить (какой режим)». Приоритизировать: сначала то, что
   ломает деньги/атрибуцию.

## Done-when
`audit.md` + `gap-list.md` записаны; `state.gap_list` заполнен с severity; каждый значимый пункт
подтверждён фактом (или явно помечен «не смог проверить — нет доступа», с указанием, что нужно от
человека). Расхождение отчёт↔источник-истины явно названо, если есть.

## Вывод пользователю
Компактно: 🟢 что работает, 🔴 что сломано/не течёт (с приоритетом), ⚪ чего нет. Предложить следующий
режим (`/redanalyst-setup` по gap-листу / `/redanalyst-ecom` / `/redanalyst-verify`).
