# /redanalyst-goals <бизнес> — какие цели и KPI ставить (ресёрч)

Вес: lightweight (research, read-only, без записи в счётчик). Отвечает на «что мерить», а не «как
настроить». Против шаблона «поставь стандартные 5 целей».

## Принцип
Цели выводятся из **воронки конкретного бизнеса**, а не по чек-листу. У доставки, B2B-услуги и
интернет-магазина разные целевые действия и разные KPI. Сначала понять бизнес, потом решить метрики.

## Шаги
1. Понять бизнес: тип (e-com / услуга / лидоген / подписка), воронка (как клиент приходит к оплате),
   где оплата фиксируется (веб / телефон / оффлайн / CRM). Спросить пробелы (gap-driven), не
   допрашивать по всему списку.
2. Лёгкий ресёрч (при необходимости): как конкуренты/ниша меряют, типовые макро/микро действия. Для
   глубокого — можно позвать `Agent` с web-поиском (не полноценный redresearch, если не просят).
3. **Спроектировать цели** (`goals-spec.md` + `state.goals` с `verified:false` — спроектированы, ещё
   не созданы в счётчике): воронка (`step`) + micro (просмотр/корзина/начал оформление) + macro (заявка/оплата). Для
   каждой — тип (`url/action/number/step/ecommerce`), условие срабатывания, где ловится (JS-событие /
   URL / офлайн). См. `references/conversion-taxonomy.md`.
4. **KPI**: под тип бизнеса — CR по этапам воронки, CPA/CPL, ROMI (если есть расход+выручка), AOV,
   доля микро→макро. Явно назвать, что из этого требует какой ноги данных (нет расхода → нет ROMI).

## Done-when
`goals-spec.md` с обоснованием (почему ИМЕННО эти цели под ЭТОТ бизнес) + список целей, готовый к
`/redanalyst-setup`. KPI привязаны к доступности данных.

## Вывод
Таблица целей (действие · тип · где ловится · micro/macro) + KPI-набор. Предложить `/redanalyst-setup`
для создания и, если оплата вне браузера, `/redanalyst-ecom` для выручки.
