# /redanalyst-crosschannel — сквозная аналитика (ROI/ROMI/CPA по кампаниям)

Вес: phased (в основном read/агрегация; write только если чинит недостающую ногу). Сводит 3 ноги в
ROI по кампаниям. См. `references/crosschannel-3legs.md`.

## Предусловие: все 3 ноги на месте
1. **Расход** — Директ связан + расход виден; не-Яндекс каналы через Costs import (`references/direct-linkage.md`).
2. **Визиты** — yclid-автопометка + консистентные UTM.
3. **Выручка** — purchase-цель verified + офлайн-конверсии `processed` (через `/redanalyst-ecom`).

Если нога отсутствует/неполна — **precondition-guard**: НЕ считать ROI как посчитанный. Явно пометить
ногу как «неизвестно/догружается» и предложить закрыть (какой режим). Ложная уверенность хуже пробела.

## Шаги
1. Проверить готовность 3 ног (из state + live-запрос). Неполнота → отчёт о том, что мешает, стоп.
2. Свести по кампаниям (Reporting API: расход `direct_client_logins` × визиты × конверсии/revenue по
   цели). Ключ склейки — визит (ClientID/yclid).
3. Посчитать `CPA`, `ROMI = (revenue − spend)/spend`, `CR` по этапам.
4. **Табличный вывод**: `кампания | расход | визиты | конверсии | выручка | CPA | ROMI`. Ячейки с
   неизвестным — `—`, догружается — `⏳`, никогда не подменять нулём.

## Done-when
Таблица ROI по кампаниям с явными состояниями ячеек; каждая нога либо подтверждена, либо явно помечена
неполной с указанием, что закрыть. Никакой посчитанный ROI на неполных данных.

## Вывод
Таблица + 2–3 вывода (какие кампании в плюс/минус, где данные неполны). Если ноги неполны — список
действий по закрытию (режимы), не «вот ваш ROI» на песке.
